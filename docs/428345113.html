Deep In Motion : 14.7 Data Recoding/Preprocessing

Created by Felix Tempel on 13.06.2025

Documentation

Preprocessing Pipeline

  • Motion Tracking: Raw video frames are input to a motion tracker (Groos et. al), which outputs per-frame joint coordinates. This is a form of recoding from raw pixel data to structured skeletal data.

  • Normalization: Coordinates may be normalized (e.g., relative to body center or scaled to a reference length) so that variations in camera distance or infant size are mitigated.

  • Temporal Windowing: Skeleton sequences are divided into fixed-length windows (e.g., 5-second windows). This recording step transforms continuous sequences into windows for the model input.

  • Feature Computation: Within each window, compute derived features (velocities, segment lengths, and angles).

  • Data Augmentation: slight rotation, scaling to skeleton data during training, specifying library or custom implementation.Â 

  • Data Formatting: Transform features into the input tensor shape required by the model/libraries (e.g., shape: [batch_size, num_time_steps, num_joints, coordinate_dims, ...]).

  • Library-Specific Requirements: see: https://github.com/DeepInMotion/JAMA2022

Justification

  • Reproducibility: By documenting each recoding step, another engineer can reproduce exactly how raw data becomes model inputs.

  • Traceability: If model performance issues arise, being able to trace back through preprocessing ensures easier debugging.

  • Regulatory compliance: Software documentation for medical devices must specify data transformations clearly, including any library-specific behaviors (e.g., version of motion-tracking library).

Regulatory References

  • ISO 13485: software design documentation; traceability of data transformations.

  • ISO 24028 (AI-specific): document preprocessing steps that affect model input distribution.
